Conceito fundamental

O que é CI/CD — em 30 segundos

CI (Integração Contínua) é a prática de rodar testes e verificações automaticamente toda vez que alguém faz um git push. Você não precisa lembrar de rodar — a máquina roda.

CD (Entrega/Deploy Contínuo) é a prática de enviar o código aprovado para produção automaticamente, sem intervenção manual.

Analogia

Imagina uma fábrica de carros. Sem CI/CD, cada operário monta seu trecho e no final alguém testa o carro inteiro — se algo quebrou, você não sabe onde nem quando. Com CI/CD, a esteira tem sensores em cada etapa: se a porta não encaixa no painel B, a linha para imediatamente e você sabe exatamente o que quebrou. Seu código é a fábrica. O GitHub Actions é a esteira com sensores.

Por que isso importa para você agora Docker sem CI/CD é um projeto pessoal. Docker com CI/CD é evidência de maturidade profissional. Todo recrutador de vaga pleno/sênior procura o badge verde no README — ele sinaliza que você entrega código confiável, não só código que funciona.

Anatomia

Dissecando um workflow YAML

Todo pipeline do GitHub Actions vive em um arquivo .yml dentro de .github/workflows/ no seu repositório. Vamos ler linha a linha:

📄 .github/workflows/ci.yml — estrutura completa anotada
name: CI Pipeline               # Nome que aparece na aba Actions do GitHub

on:                             # QUANDO este workflow deve rodar?
  push:
    branches: [main]           # Roda a cada push na branch main
  pull_request:
    branches: [main]           # E a cada PR aberto para main

jobs:                            # O QUE fazer? (pode ter vários jobs em paralelo)
  build-and-test:               # Nome do job (você escolhe)
    runs-on: ubuntu-latest     # Em qual máquina rodar (grátis no GitHub)

    steps:                       # Passos sequenciais dentro do job

      - name: Checkout código
        uses: actions/checkout@v4  # Action oficial: clona seu repositório

      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'  # Instala o .NET na VM

      - name: Restaurar dependências
        run: dotnet restore          # Equivale ao npm install

      - name: Build
        run: dotnet build --no-restore --configuration Release

      - name: Testes
        run: dotnet test --no-build --verbosity normal

Esse é o arquivo mínimo para um projeto .NET. Cada linha tem um propósito claro — nada de magia.

Vocabulário

Os 6 conceitos que você precisa dominar

Workflow

O arquivo YAML inteiro. Um repositório pode ter vários workflows (ex: um para CI, outro para deploy).

Event (on:)

O gatilho. Pode ser push, pull_request, schedule (cron), workflow_dispatch (manual), e mais.

Job

Um grupo de steps que roda em uma máquina. Múltiplos jobs rodam em paralelo por padrão.

Step

Um comando ou action dentro de um job. Sempre executa em sequência dentro do job.

Action

Um passo reutilizável publicado no marketplace. Ex: actions/checkout é mantido pelo próprio GitHub.

Runner

A máquina virtual onde o job roda. O GitHub oferece Ubuntu, Windows e macOS — grátis para repositórios públicos.

Segurança

Segredos e variáveis de ambiente

Você nunca coloca senhas, tokens ou connection strings diretamente no YAML. O GitHub tem um cofre chamado Secrets — você armazena lá e referencia no workflow:

📄 Usando secrets no workflow
      - name: Deploy
        env:
          CONNECTION_STRING: ${{ secrets.DB_CONNECTION_STRING }}
          API_KEY: ${{ secrets.OPENAI_KEY }}
        run: dotnet publish

Para cadastrar: Repositório → Settings → Secrets and variables → Actions → New repository secret. O valor nunca aparece nos logs — o GitHub mascarara automaticamente.

Regra de ouro Se o valor muda entre ambientes (dev/prod) ou é sensível: vai para Secrets. Se é público e fixo (versão do .NET, nome do projeto): vai direto no YAML.
Integração com Aula 1

Docker dentro do GitHub Actions

Você aprendeu Docker na Aula 1. Agora o CI/CD vai usar seu Docker para garantir que o build funciona em qualquer ambiente — incluindo o servidor de produção.

📄 Workflow que builda e publica imagem Docker
name: Docker Build & Push

on:
  push:
    branches: [main]

jobs:
  docker:
    runs-on: ubuntu-latest
    steps:

      - uses: actions/checkout@v4

      - name: Login no Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_TOKEN }}

      - name: Build e Push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: enzoagw/nexus-core:latest

Isso significa: a cada push na main, o GitHub builda a imagem Docker do nexus-core e publica automaticamente no Docker Hub. Qualquer pessoa pode rodar com docker pull enzoagw/nexus-core.

Como implementar no nexus-core agora

1

Criar a estrutura de pastas

No raiz do repositório nexus-core, crie a pasta .github/workflows/ e o arquivo ci.yml.

2

Use o arquivo da seção "Anatomia" acima. Ajuste a versão do .NET para a que você usa no projeto.

Com o pipeline verde, cole o badge no README e faça um post no LinkedIn: "Configurei CI/CD no nexus-core com GitHub Actions."

Armadilhas

Erros que todo mundo comete na primeira vez

Erro Causa Solução
Indentação YAML inválida YAML usa espaços, não tabs — e o número de espaços importa Use o validador online yaml.lint ou instale a extensão YAML no VS Code
dotnet restore falha Versão do .NET no workflow diferente da do projeto Verifique o global.json ou o .csproj e ajuste o dotnet-version
Testes não encontrados dotnet test na raiz não encontra os projetos de teste Aponte o caminho: dotnet test ./NexusCore.Tests/
Secret não funciona Nome do secret no GitHub diferente do que está no YAML Case-sensitive: secrets.DB_STRING ≠ secrets.db_string
Workflow não dispara Branch errada no on.push.branches Verifique se sua branch principal é main ou master
Próximo nível

O que vem depois (não agora)

Foco desta semana é o pipeline básico com badge verde. Mas quando quiser expandir:

Matrix builds

Rodar o mesmo job em múltiplas versões do .NET ou sistemas operacionais simultaneamente.

Caching de dependências

Salvar o resultado do dotnet restore entre runs — reduz tempo de 3 min para 30s.

Ambientes (Environments)

Separar staging de production com aprovação manual antes do deploy em prod.

Artifacts

Salvar o binário compilado ou relatório de cobertura de testes entre jobs.
