Chapter 1 · The Problem

Why does Docker exist?

Before you can understand what Docker is, you need to grasp the problem it solves. Without that understanding, it seems like an unnecessarily complex tool.

The phrase that killed projects for decades.

"It works on my machine."

You finish a feature, it works perfectly locally. You deploy it to the client's server and it breaks. Why? Because the server has a different version of Node. Or .NET. Or an environment variable with a different name. Or a different operating system. Or a database with a different configuration.

The environment where the code runs matters just as much as the code itself. And environments are fragile, inconsistent, and impossible to "version" the traditional way.

What Docker solves

Docker creates a complete package that contains not just your code, but the entire environment it needs to run: the base operating system, the dependencies, the configurations, the environment variables. Everything together, versioned, and portable.

Docker's promise: if it runs on your machine, it runs anywhere. No exceptions. The environment travels with the code.

Think of it this way: instead of delivering only your project's code and hoping the server has the right version of .NET, you deliver the project plus .NET plus the configurations plus the required operating system, all packaged into a single file.

Chapter 2 · The Analogy

The right analogy for a developer

There are several analogies for Docker. The most common one is the recipe vs. the dish. It works, but it is superficial. Here is the analogy that makes the most sense for someone who programs.

An Image is a Class. A Container is an Instance.

📋

Image

It is the definition. A static file that describes what the environment should look like. It does not run. It does not consume memory. It exists only as a blueprint.

→
⚡

Container

It is the execution. An image that has been "instantiated" and is running as a live process, with its own memory, CPU, and state.

Exactly like in OOP:

analogy.ts
// IMAGE = the class definition (static, does not run) class MyApi { // .NET 8 runtime // application files // environment configurations } // CONTAINER = instances running in memory const container1 = new MyApi() // port 8080 const container2 = new MyApi() // port 8081 const container3 = new MyApi() // port 8082 // The CLASS does not change. Instances have their own state and life cycle.

You can create as many containers as you want from the same image. Each one will have its own state, its own memory, its own life cycle — but all of them start from the same definition.

Golden rule: the image is immutable. You never "edit" a running image. You edit the Dockerfile, generate a new image, and create new containers from it.
Chapter 3 · Image

What is an Image — in depth

A Docker image is a binary file composed of stacked layers. Each layer represents one instruction from the Dockerfile. Understanding this is fundamental to using Docker efficiently.

The Dockerfile — the recipe that generates the image

An image does not appear out of thin air. You describe it in a file called Dockerfile. Each line of that file becomes a layer of the image.

Dockerfile
# Layer 1 — which base operating system to use FROM mcr.microsoft.com/dotnet/aspnet:8.0 # Layer 2 — set the working directory inside the container WORKDIR /app # Layer 3 — copy your project files into the image COPY . . # Layer 4 — expose the port the application uses EXPOSE 8080 # Layer 5 — the command that runs when the container starts ENTRYPOINT ["dotnet", "MyApi.dll"]

Layers — what makes Docker smart

Each Dockerfile instruction creates a layer. Docker caches each layer separately. If you only change your code (Layer 3), it does not need to download .NET again (Layer 1) or recreate the directory (Layer 2) — it only re-executes from the modified layer onward.

5
ENTRYPOINT — startup command
4
EXPOSE — application port
3
COPY — your source code
2
WORKDIR — working directory
1
FROM — base operating system

All these layers are read-only. The image itself never changes. When you create a container from it, Docker adds a single extra layer on top — the write layer.

Where do images live?

Images are stored in registries. The default registry is Docker Hub — think of it as npm, but for Docker images. When you run docker pull nginx, you are downloading the official nginx image from Docker Hub to your machine.

Docker Hub

Public registry. Official images for nginx, postgres, node, .NET, and more. Anyone can publish and pull.

Private registry

Your company can have its own registry (AWS ECR, Azure ACR, GitHub Container Registry) for internal images.

Local cache

Images you pull or build are cached on your machine. docker images lists all of them.

Tags

Versions of an image. nginx:latest, node:20, postgres:16-alpine. Always specify the version in production.

Chapter 4 · Container

What is a Container — in depth

A container is a running image. But understanding what happens under the hood changes how you work with them.

A container is not a virtual machine

This is the biggest misconception about Docker. Many people think container = lightweight VM. It is not.

Characteristic Virtual Machine (VM) Docker Container
Operating system Its own, complete (kernel + everything) Shares the host kernel
Startup time Minutes Milliseconds to seconds
Typical size GBs MBs
Isolation Complete (virtualized hardware) Isolated process (namespaces + cgroups)
Performance overhead Significant Near zero

A container is, at its core, an isolated process running on your operating system. Linux uses two features to create this isolation:

Namespaces

Make the container see only its own processes, network, and file system — as if it were its own OS, but it is not.

cgroups

Limit the resources the container can use: CPU, memory, I/O. This ensures one container cannot consume everything on the host machine.

The write layer — why containers are ephemeral

Remember that all image layers are read-only? When a container is created, Docker adds a temporary write layer on top.

✏
Write layer (container) — temporary, destroyed when stopped
R
Layer 5 — read-only (from the image)
R
Layer 4 — read-only (from the image)
R
Layer 3 — read-only (from the image)
Important implication: any data you write inside a container (logs, files, database data) is lost when the container stops. To persist data, you use volumes — we will learn that next week.

Container states

A container has a life cycle. Understanding the states is essential for debugging problems.

Container life cycle

created docker create
↓ docker start
running active process
docker pause
paused frozen
docker stop
exited stopped, still exists
↓ docker rm
removed destroyed
Crucial detail: docker stop stops the container but does not destroy it. It still exists with its state intact. docker rm destroys it permanently. Use docker run --rm to destroy it automatically when it stops.

Chapter 6 · Commands

The commands you will use every day

Images

terminal
# Pull an image from Docker Hub $ docker pull nginx # List all local images $ docker images REPOSITORY TAG IMAGE ID CREATED SIZE nginx latest a72860cb95fd 2 weeks ago 187MB # Build an image from the Dockerfile in the current directory $ docker build -t my-api:v1 . # ↑ name:tag ↑ context (current directory) # Remove an image $ docker rmi my-api:v1

Containers

terminal
# Create and start a container $ docker run nginx # Most important flags for docker run: $ docker run \ -d \ # detached: runs in the background (does not block the terminal) -p 8080:80 \ # port: maps host:container --name my-nginx \ # friendly name --rm \ # automatically removes the container when it stops nginx # image name # List running containers $ docker ps # List ALL containers (including stopped ones) $ docker ps -a # View logs from a container $ docker logs my-nginx $ docker logs -f my-nginx # -f = follow (real-time) # Open a terminal inside a running container $ docker exec -it my-nginx bash # ↑ -i = interactive, -t = terminal # Stop a container $ docker stop my-nginx # Remove a stopped container $ docker rm my-nginx # Stop AND remove in one command $ docker rm -f my-nginx

The -p flag explained

The -p 8080:80 flag is the one that confuses people the most at first. It maps ports:

-p [your-machine-port]:[port-inside-the-container]

The container lives on an isolated network. It has no idea there is an "outside." You need to create a bridge: "when someone knocks on port 8080 of my machine, forward it to port 80 inside the container."
Chapter 7 · Goal of the Day

Hands-on — container running in the terminal

Now let's execute. Follow each step in order. The goal is to have a container running and see the result in your browser.

Prerequisite

Docker Desktop installed and running. If you have not installed it yet: docker.com/products/docker-desktop

1

Verify Docker is running

Open the terminal and run:

PowerShell
$ docker --version Docker version 26.1.1, build 4cf5afa $ docker ps CONTAINER ID IMAGE COMMAND CREATED STATUS PORTS NAMES # If that header line appears, Docker is working.
2

Run your first container — nginx

Let's start an nginx web server. Docker will download the image automatically if it is not available locally.

PowerShell
$ docker run -d -p 8080:80 --name my-nginx nginx # What each part means: # -d → runs in the background (does not block the terminal) # -p 8080:80 → maps port 8080 on your machine → port 80 inside the container # --name → gives the container a friendly name # nginx → name of the image to use Unable to find image 'nginx:latest' locally latest: Pulling from library/nginx ... a72860cb95fd2b0f4a... ← ID of the created container
3

Confirm it is running

PowerShell
$ docker ps CONTAINER ID IMAGE STATUS PORTS NAMES a72860cb95fd nginx Up 3 seconds 0.0.0.0:8080->80/tcp my-nginx
4

Open it in the browser

Go to http://localhost:8080 — you will see the default nginx page. A complete web server running in an isolated container on your machine.

5

Explore — enter the container

PowerShell
# Open a terminal inside the container (like SSH) $ docker exec -it my-nginx bash root@a72860cb95fd:/# ← you are now INSIDE the container # See the processes that exist INSIDE this container root@a72860:/# ps aux USER PID COMMAND root 1 nginx: master process nginx 29 nginx: worker process # Only these processes exist here. Total isolation. # Exit the container root@a72860:/# exit
6

View logs in real time

PowerShell
# Watch the logs while you visit localhost:8080 in the browser $ docker logs -f my-nginx 172.17.0.1 - - [14/May/2026] "GET / HTTP/1.1" 200 615 172.17.0.1 - - [14/May/2026] "GET /favicon.ico" 404 153
7

Stop and clean up

PowerShell
$ docker stop my-nginx my-nginx $ docker rm my-nginx my-nginx $ docker ps CONTAINER ID IMAGE COMMAND CREATED STATUS PORTS NAMES # Empty — all clean.
✅

Goal of the day achieved

You ran a container, explored its interior, watched logs in real time, and removed it correctly. You now understand the real difference between an image and a container.

Summary

What you learned today

  • Docker solves the "it works on my machine" problem by packaging code and environment together
  • Image = static definition (like a class). Immutable, versionable, portable
  • Container = execution of an image (like an instance). Has its own state and its own life cycle
  • Images are composed of cached layers — builds are fast because only the changed layer is re-executed
  • Containers share the host kernel — they are not VMs, they are isolated processes
  • The container's write layer is temporary — data is lost when it stops (volumes solve this)
  • Essential commands: docker run, docker ps, docker stop, docker rm, docker logs, docker exec
  • The -p flag maps a port on your machine to a port inside the container
