Persistência Volátil vs Permanente:
Engenharia de Volumes

Foco Prático e Mecânico: Dominar o ciclo de vida dos dados e os mecanismos de armazenamento permanente para Bancos de Dados e Aplicações .NET Core em produção.
A Nuance dos Dados

Por que Containers perdem dados?

Como vimos na Aula 1, um container adiciona uma camada fina de escrita (Read-Write Layer) no topo das camadas somente leitura da imagem. O problema técnico central é: esta camada está diretamente amarrada ao ciclo de vida do container físico.

Se o processo cair, os dados continuam lá (retidos no container parado). Porém, se você executar um docker rm para atualizar a versão da sua API ou banco de dados, essa camada de escrita é sumariamente deletada do sistema de arquivos do Host.

Para resolver isso, o Docker fornece duas estratégias principais de desacoplamento de storage:

  • Named Volumes (Volumes Nomeados): O Docker gerencia uma pasta oculta no sistema hospedeiro (em /var/lib/docker/volumes/) com isolamento e performance otimizados. Ideal para Bancos de Dados (SQL Server, PostgreSQL, MongoDB).
  • Bind Mounts (Montagens de Vínculo): Você mapeia explicitamente uma pasta exata do seu computador (ex: C:\projects\logs) para dentro do container. Ideal para ambientes de desenvolvimento local onde você quer ver arquivos mudando em tempo real.
🏢 Visão de Mercado (Pleno/Sênior): Erros de montagem de volumes são causas clássicas de perda catastrófica de dados em ambientes produtivos. Engenheiros de software de elite nunca armazenam estados persistentes dentro de camadas voláteis de containers.
Mão na Massa Atômica

Laboratório Prático: Comandos de Persistência

Siga estritamente o fluxo estruturado de Comando -> Verificação -> Validação de Impacto.

Passo 1: Provocando o Erro (Cenário Sem Volume)

Vamos criar um container temporário e simular a criação de um arquivo crítico dentro dele.

Terminal / PowerShell
$ docker run --name banco-volatil -d alpine sh -c "echo 'dados_secretos_de_producao' > /tmp/db.txt && sleep 3600"

Agora, vamos verificar se o arquivo existe usando o comando de inspeção interna:

Terminal / PowerShell
$ docker exec banco-volatil cat /tmp/db.txt

⚡ Micro-Exercício 1: A Simulação da Catástrofe

Execute a deleção forçada deste container simulando uma atualização de ambiente ou crash:

$ docker rm -f banco-volatil

Agora tente subir um novo container com o mesmo nome e ler o arquivo de novo. O que acontece? Os dados sumiram permanentemente porque o storage era volátil.

Passo 2: Engenharia com Named Volumes

Agora faremos a abordagem correta de nível empresarial, criando e acoplando um volume nomeado independente.

Terminal / PowerShell
$ docker volume create meu_banco_dados_prod $ docker volume ls

Vamos instanciar o container montando esse volume explicitamente na pasta de dados dele usando a flag -v ou --mount:

Terminal / PowerShell
$ docker run -d --name banco-persistente -v meu_banco_dados_prod:/data alpine sh -c "echo 'dados_salvos_no_volume_seguro' > /data/seguranca.db && sleep 3600"

⚡ Micro-Exercício 2: A Prova Prática da Resiliência

Execute os seguintes passos em ordem:

  1. Delete o container agressivamente: docker rm -f banco-persistente
  2. Suba um novo container totalmente novo usando o MESMO volume:
    docker run --name banco-novo -v meu_banco_dados_prod:/data alpine cat /data/seguranca.db
  3. Observe o resultado no terminal. O arquivo foi lido com sucesso! O dado sobreviveu à destruição completa da unidade computacional.

Passo 3: Bind Mounts na Prática (Hot-Reload de Configurações)

Vamos expor um arquivo HTML local direto para dentro do Nginx sem precisar reconstruir a imagem do container.

⚡ Micro-Exercício 3: Mapeamento Direto de Arquivos

1. Crie uma pasta vazia em qualquer local do seu computador e crie um arquivo chamado index.html dentro dela com um texto qualquer.

2. Rode o comando abaixo, substituindo /caminho/da/sua/pasta pelo caminho absoluto da pasta que você acabou de criar:

$ docker run -d -p 8085:80 -v /caminho/da/sua/pasta:/usr/share/nginx/html --name site-local nginx

3. Acesse http://localhost:8085. Altere o texto do arquivo no seu VS Code, salve e atualize o navegador. O arquivo reflete instantaneamente!

Consolidação e Retenção

Revisão Ativa Baseada em Evidências

Fixe os conceitos respondendo mentalmente a este checklist arquitetural antes de avançarmos para a próxima aula:

1. Quando uso Named Volume vs Bind Mount?

Named Volumes para persistência de dados internos cruciais que a aplicação gerencia (ex: Bancos de dados em Prod). Bind Mounts quando arquivos locais precisam interagir com o container (ex: Código fonte ou arquivos de configuração em Dev).

2. O comando 'docker rm' limpa volumes?

Não. Os volumes são cidadãos de primeira classe no Docker. Eles possuem ciclo de vida totalmente independente e só deixam de existir se você rodar explicitamente docker volume rm [NOME] ou docker volume prune.

3. Qual o perigo de usar Bind Mounts em Produção multi-node?

Bind Mounts dependem da estrutura exata de caminhos de arquivos do Host local. Se você migrar para um cluster de servidores (Kubernetes), o outro servidor pode não ter a pasta no mesmo caminho, quebrando o deploy imediatamente.

4. O que faz o comando 'docker exec'?

Ele permite que você execute um processo extra ad-hoc dentro de um container que já está ativo em background, sendo a principal ferramenta de debug técnico do desenvolvedor.
